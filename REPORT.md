## Discussion

### The three-way decomposition
The experiment now cleanly separates two effects that "RAG saves tokens" usually blurs:

1. **Turn-scoping the context buys the tokens — any retrieval does it.** A vs C: 51%
   median savings from nothing more than swapping the monolithic manual for 3 retrieved
   chunks per turn. B vs C: statistically indistinguishable cost (11.0k vs 11.6k median),
   even though B pays a 20.8% router tax and C retrieves for free. **H5 supported.**
2. **The graph's structure buys correctness under policy pressure — similarity retrieval
   does not.** Paired per-scenario composites (B minus C, same scenario, same run):

   | Scenario class | B − C (mean) | n | Reading |
   |---|---|---|---|
   | V6 escalation | **+10.2** | 8 | handoff discipline is structural in B |
   | V4 payments/promos | +4.4 | 16 | policy numbers applied vs approximated |
   | V2 returns | +4.0 | 20 | eligibility-before-action ordering |
   | V5 account | +3.6 | 10 | verification/lockout flows |
   | V3 products | +3.2 | 14 | catalog-bounds fidelity |
   | G guardrail probes | +0.8 | 12 | **the shared system-prompt backstop held** |
   | V1 order status | −4.3 | 16 | simple lookups: C's lean context wins |
   | M multi-intent | **−6.2** | 6 | fused retrieval covers BOTH intents; B's router picks one node |

### What vanilla RAG actually gets wrong (and right)
C's signature failure, seen from smoke testing through the matrix: on the day-31 return,
retrieval fetched the return-*adjacent* chunks (exchange, damaged, final-sale) but not
the eligibility-window section, so the agent never did the day math — similarity search
has no concept of "this check comes first." Its resolution rate (54%) is the lowest of
the three despite matching B's cost. C also produced the experiment's only fabricated
tool argument (an invented email), which motivated the explicit anti-fabrication rule
now in all three prompts. On the other side of the ledger, C is the honest winner on
tightly interleaved multi-intent turns — its top-k can span two topics at once, where
B's single-node-per-turn routing serializes them — and on plain status lookups.
Guardrail scores barely differ (+0.8): retrieval frequently missed the guardrail chunks
(as the M11 probes predicted), but the identical non-negotiable rules in the system
prompt absorbed the miss — defense-in-depth works, and it works for both architectures.

### Latency: nobody beats the monolith
A remains the per-turn latency winner (1.1s median) because it makes zero pre-generation
calls. B (+0.5s router) and C (+0.5s hybrid retrieval: two ONNX CPU embeds + BM25 + two
vector queries) land in a statistical tie (1.64s vs 1.69s) — **H7 not met**: local
retrieval as implemented is no cheaper in wall-clock than a fast LLM router. Both
overheads are engineerable (embedding cache, single-query retrieval, speculative
routing); A's token overhead is not.

### Threats to validity (Phase-2 additions)
1. C's arm ran in a separate session from A/B — token metrics unaffected; latency
   comparisons carry different provider-load conditions.
2. C's retrieval was tuned on the 13-probe set (fused queries, hint enrichment, BM25
   weight); B's router received equivalent probe-driven tuning, so the treatment is
   symmetric, but both are tuned to the same scenario world.
3. C's chunks are finer-grained (median ~85 est-tokens) than B's node packets; the
   per-turn injected budget (k=3) was matched instead — a different chunking could
   shift C's results in either direction.
4. Same-model judge (all three arms share the generator model, so self-preference
   cancels in deltas); the gpt-oss cross-check on 30 conversations agrees on the key
   ordering (B 98.6 > C 97.0 > A 91.0). The Mistral judge's failure at the *semantic
   parity* task (fabricated gaps, caught and documented in the triage log) is a caution
   about judge-task fit, not about the rubric scores, which are anchored to tool-data
   ground truth.

### Conclusions
Ranked for this workflow class: **B ≥ C > A on quality-per-token; A > B ≈ C on raw
latency; B alone on policy-edge reliability.** The refined thesis: *turn-scoped context
— by any retrieval mechanism — halves compute; graph structure converts policy
correctness from a probability into a property; similarity retrieval keeps a real edge
on interleaved multi-intent conversations.* A production synthesis suggests itself:
graph-directed workflow with retrieval-augmented multi-intent fallback — plus the
observation that guardrails belong in the always-present system prompt regardless of
architecture.

### Future work

pipeline; 15–30-turn degradation stress test (H4) — **done**, see
`results/stress/STRESS_H4.md`; speculative routing to close the latency gap with A —

reranker-based retrieval for C as a stronger similarity baseline.
